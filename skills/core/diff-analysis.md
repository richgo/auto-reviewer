# Core Skill: Diff Analysis

## Purpose
Parse, understand, and contextualize PR diffs to prepare for specialized review. Identifies changed files, affected functions, added/removed/modified lines, and filters out noise (generated code, renames, formatting). Provides structured diff data to downstream review skills.

## Scope
1. **Parse diff format** — understand unified diff, git diff, GitHub/GitLab PR diff format
2. **Detect change types** — add, modify, delete, rename, move
3. **Extract context** — affected functions, classes, surrounding code
4. **Filter noise** — generated files, lock files, formatting-only changes
5. **Identify review areas** — which files need deep review vs surface scan

## Diff Format Understanding

### Unified Diff Format
```diff
diff --git a/file.py b/file.py
index abc123..def456 100644
--- a/file.py
+++ b/file.py
@@ -10,7 +10,8 @@ def process_user(user_id):
 def login(username, password):
-    if check_password(username, password):
+    hashed = hash_password(password)
+    if check_password(username, hashed):
         return create_session(username)
     return None
```

**Key elements:**
- `diff --git` — file being changed
- `---/+++` — old/new file paths
- `@@` — line range (`-old_start,old_count +new_start,new_count`)
- `-` — removed line
- `+` — added line
- (no prefix) — context line

### Change Types

**1. Addition:**
```diff
+new_function()
```

**2. Deletion:**
```diff
-old_function()
```

**3. Modification (delete + add):**
```diff
-x = 10
+x = 20
```

**4. Rename/Move (no content change):**
```diff
diff --git a/old_path.py b/new_path.py
similarity index 100%
rename from old_path.py
rename to new_path.py
```

**5. Rename + Modify:**
```diff
diff --git a/old_path.py b/new_path.py
similarity index 95%
rename from old_path.py
rename to new_path.py
@@ -5,3 +5,4 @@
 existing_code()
+new_code()
```

## Detection Strategy

### Step 1: Identify Files
Extract list of changed files with change type:
- **Added:** New files (`new file mode`)
- **Deleted:** Removed files (`deleted file mode`)
- **Modified:** Changed files (has `@@` hunks)
- **Renamed:** Moved files (`rename from/to`)

**Output:**
```json
{
  "files": [
    {"path": "src/auth.py", "type": "modified", "additions": 5, "deletions": 2},
    {"path": "src/payment.py", "type": "added", "additions": 120, "deletions": 0},
    {"path": "old/utils.py", "type": "renamed", "new_path": "src/utils.py"}
  ]
}
```

### Step 2: Extract Changed Lines
For each file, extract:
- **Added lines** with line numbers
- **Deleted lines** with line numbers
- **Context lines** (surrounding unchanged code)

**Output:**
```json
{
  "file": "src/auth.py",
  "changes": [
    {
      "hunk": "@@ -10,7 +10,8 @@",
      "function": "login",
      "added": [
        {"line": 12, "content": "    hashed = hash_password(password)"},
        {"line": 13, "content": "    if check_password(username, hashed):"}
      ],
      "deleted": [
        {"line": 11, "content": "    if check_password(username, password):"}
      ]
    }
  ]
}
```

### Step 3: Determine Function/Class Context
Parse the diff context lines to identify:
- **Function name** (from `@@` line or context)
- **Class name** (if inside class)
- **Module/namespace**

**Heuristics:**
- `@@ ... @@ def function_name(...)` — Python function
- `@@ ... @@ func FunctionName(...)` — Go function
- `@@ ... @@ public void methodName(...)` — Java method
- `@@ ... @@ class ClassName` — Class definition

### Step 4: Filter Noise

**Generated files to ignore:**
- `package-lock.json`, `yarn.lock`, `Gemfile.lock`, `Pipfile.lock`, `poetry.lock`
- `*.min.js`, `*.min.css` (minified files)
- `dist/`, `build/`, `target/` directories
- `*.pb.go` (generated protobuf)
- `*.g.dart` (generated Dart)
- Files with `// Generated by` or `# AUTO-GENERATED` headers

**Formatting-only changes:**
- Whitespace-only changes (can be detected with `git diff -w`)
- Import sorting
- Line ending changes (CRLF ↔ LF)

**Low-priority changes:**
- Test file updates (review but lower priority than production code)
- Documentation changes (README, comments)
- Configuration changes (unless security-related)

### Step 5: Classify Review Priority

**High priority (deep review required):**
- Security-sensitive files (auth, crypto, payment, database queries)
- New files (complete review needed)
- Core business logic
- Public API changes

**Medium priority:**
- Modified utility functions
- Refactoring (function rename, extract method)
- Test file changes

**Low priority:**
- Formatting changes
- Comment updates
- README changes
- Generated file updates (review for unintended inclusions)

**Output:**
```json
{
  "high_priority": ["src/auth.py", "src/payment.py"],
  "medium_priority": ["src/utils.py", "tests/test_auth.py"],
  "low_priority": ["README.md"],
  "ignored": ["package-lock.json", "dist/bundle.min.js"]
}
```

## Rename Detection

### Scenario: File Renamed + Modified
```diff
diff --git a/old/user_manager.py b/new/user_service.py
similarity index 92%
rename from old/user_manager.py
rename to new/user_service.py
@@ -15,6 +15,7 @@ def create_user(username, email):
     user = User(username, email)
     db.save(user)
+    send_welcome_email(user.email)
     return user.id
```

**Detection logic:**
1. **Similarity index >80%** → mostly rename
2. **Has rename from/to** → definitely rename
3. **Small diff** (< 10% of file) → focus review on actual changes
4. **Extract only changed portions** for review (ignore unchanged code)

## Generated File Detection

### Heuristics
```python
def is_generated_file(file_path, first_lines):
    # Check file path
    generated_paths = [
        'node_modules/', 'vendor/', 'dist/', 'build/', 'target/',
        '.lock', '.min.js', '.min.css', '.pb.go', '.g.dart'
    ]
    if any(pattern in file_path for pattern in generated_paths):
        return True
    
    # Check file header
    generated_markers = [
        '// AUTO-GENERATED',
        '# Generated by',
        '// Code generated by',
        '/* eslint-disable */',
        '// @generated'
    ]
    for line in first_lines[:5]:
        if any(marker in line for marker in generated_markers):
            return True
    
    return False
```
